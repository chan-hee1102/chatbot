## 1. 문서 개요

### 1.1 목적
본 문서는 웹 기반 LLM 챗봇 서비스의 비즈니스 목표, 기능 비전, 시스템 구조, UX 원칙, 운영 전략을 정의합니다. 개발팀·운영팀·이해관계자가 동일한 기준을 공유하기 위한 단일 참조 문서(Single Source of Truth)로 기능합니다.

### 1.2 범위
| 구분 | 내용 |
|---|---|
| 포함 | 회원가입·로그인, 멀티 대화 세션, 실시간 스트리밍 응답, 대화 영속화, 마크다운 렌더링, 반응형 UI, 단일 클라우드 배포 |
| 제외 | 파일 업로드, 이미지/음성 입출력, 소셜 로그인, 결제, 어드민 콘솔 |
| 확장 후보 | 멀티 LLM 선택, 사용자별 시스템 프롬프트, 사용량 대시보드, 팀 워크스페이스 |

### 1.3 용어 정의
| 용어 | 정의 |
|---|---|
| 대화 세션(Conversation) | 하나의 독립된 채팅방. 사용자당 N개 보유 |
| SSE | Server-Sent Events. 서버 → 클라이언트 단방향 실시간 푸시 |
| JWT | JSON Web Token. 무상태 인증 토큰 |
| LLM | Large Language Model. 본 서비스는 Google Gemini 2.5 Flash 사용 |
| Context Window | 모델 호출 시 함께 전송되는 직전 대화 이력 범위 |
| Standalone Output | Next.js의 컨테이너 배포 최적화 빌드 산출물 |

## 2. 서비스 개요

### 2.1 배경
범용 AI 챗봇 서비스는 대화 이력의 영구 보존, 자체 UX 커스터마이징, 사용량 통제, 데이터 주권 측면에서 한계가 있습니다. 본 서비스는 개인이 직접 운영하는 전용 챗봇 인프라를 구축해 **이력의 완전 소유**와 **확장 가능한 베이스라인**을 동시에 확보합니다.

### 2.2 비전
> "내가 통제하는 인프라 위에서, ChatGPT 수준의 대화 경험을 누구나 자신의 데이터로 누린다."

### 2.3 핵심 가치 제안
| 가치 | 사용자 관점 |
|---|---|
| **연속성** | 모든 대화가 영구 저장되어 언제든 다시 열람·이어가기 가능 |
| **개인화** | 사용자별 독립 워크스페이스, 향후 시스템 프롬프트 커스터마이징 |
| **즉시성** | 토큰 단위 스트리밍으로 평균 첫 응답 2초 이내 체감 지연 |
| **확장성** | 단일 사용자 → 다중 사용자 → 멀티 모델로의 점진적 확장 경로 보유 |
| **신뢰성** | HttpOnly 쿠키·bcrypt·Prepared Statement로 보안 기본기 충족 |

### 2.4 성공 지표 (KPI)
| 지표 | 목표값 | 측정 주기 |
|---|---|---|
| 가입→첫 메시지 전환율 | 70% 이상 | 주간 |
| 첫 토큰 수신 평균 시간(TTFT) | 2.0초 이내 | 일간 |
| 정상 응답 비율 (HTTP 5xx 제외) | 99% 이상 | 일간 |
| 월간 활성 사용자(MAU) 재방문율 | 40% 이상 | 월간 |
| 평균 세션당 메시지 수 | 8개 이상 | 주간 |

## 3. 타겟 사용자

### 3.1 1차 타겟
**개발자/지식 노동자 (1인 사용)**
- 본인의 질문·답변·코드 스니펫을 영구 보존하고 검색 가능 자산으로 축적

## 4. 핵심 기능

### 4.1 기능 맵
```
인증            │ 회원가입 · 로그인 · 로그아웃
                │
대화 관리        │ 새 대화 · 목록 · 자동 제목 · 삭제(Cascade)
                │
채팅            │ 메시지 전송 · SSE 스트리밍 · 응답 중단
                │ 대화 이력 영속화 · 컨텍스트 윈도우 관리
                │
렌더링          │ 마크다운 · 코드 하이라이팅 · 표 · 인라인 코드
                │
시스템          │ 시스템 프롬프트 자동 주입 (현재 시각 KST + 페르소나)
                │
UX              │ 반응형 (모바일 768px) · 라이트 테마
                │ 응답 대기 도트 애니메이션 · 스트리밍 커서
                │ 버튼 hover lift/pop 마이크로 인터랙션
```

### 4.2 핵심 사용자 시나리오
| # | 시나리오 | 주요 흐름 |
|---|---|---|
| S1 | 신규 가입 후 첫 대화 | `/register` → 자동 로그인 → `/chat` → 새 대화 생성 → 첫 메시지 전송 → 자동 제목(앞 20자) → 사이드바 갱신 |
| S2 | 기존 대화 이어가기 | `/login` → `/chat` → 사이드바에서 과거 세션 클릭 → 메시지 전체 로드 → 추가 메시지 입력 |
| S3 | 긴 응답 중단 | 응답 스트리밍 중 "중단" 버튼 → AbortController로 클라이언트 중단 → 부분 응답을 "_(중단됨)_" 표기와 함께 DB 저장 |
| S4 | 대화 정리 | 사이드바에서 휴지통 클릭 → 확인 → DB cascade 삭제 → 라우터 이동 (`/chat`으로) |
| S5 | 모바일 사용 | 햄버거 메뉴 → 드로어 슬라이드 인 → 대화 선택 → 자동 닫힘 |

## 5. 시스템 아키텍처

### 5.1 컴포넌트 구성도

```
┌────────────────────────────────────────────────────────────────┐
│                        Client (Browser)                         │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  React Components (App Router, RSC + Client Components)  │  │
│  │  - AuthForm · Sidebar · ChatWindow · MessageBubble       │  │
│  │  - Hooks: useConversations · useChat (SSE consumer)      │  │
│  └──────────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────────┘
                 │  HTTPS  ▲   SSE (text/event-stream)
                 ▼         │
┌────────────────────────────────────────────────────────────────┐
│       Next.js 14 (App Router) on Railway (standalone)           │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  Edge Middleware: JWT verify (jose) → x-user-id header   │  │
│  └──────────────────────────────────────────────────────────┘  │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  Route Handlers (Node runtime)                            │  │
│  │  /api/auth · /api/conversations · /api/messages · /api/chat│ │
│  └────────────┬─────────────────────────────────┬───────────┘  │
│               │                                 │              │
│               ▼                                 ▼              │
│      ┌───────────────────┐         ┌────────────────────────┐ │
│      │ better-sqlite3    │         │  @google/generative-ai │ │
│      │ Prepared Stmts    │         │  SSE → TransformStream │ │
│      └────────┬──────────┘         └────────────┬───────────┘ │
└───────────────┼────────────────────────────────┼──────────────┘
                ▼                                ▼
        ┌───────────────┐                ┌────────────────┐
        │ SQLite (WAL)  │                │  Gemini API    │
        │ Volume Mount  │                │  (Google AI)   │
        │ /app/data     │                └────────────────┘
        └───────────────┘
```

### 5.2 메시지 전송 데이터 플로우
```
[User Input]
   │
   ▼
POST /api/chat  {conversationId, content}
   │
   ├─ JWT verify (middleware)
   ├─ Save user message → SQLite
   ├─ If first message: auto-title (first 20 chars)
   ├─ Load last 20 messages (context window)
   ├─ Compose systemInstruction (KST time + persona)
   │
   ▼
Gemini API · streamGenerateContent
   │  (token-by-token)
   ▼
TransformStream → SSE:  data: {text}\n\n  ...  data: [DONE]\n\n
   │
   ├─ Client parses SSE → setDraft(accumulated)
   ├─ On finish: append model message, clear draft
   └─ Server-side: save full model message, touch updated_at
```

### 5.3 인증·세션 흐름
1. **회원가입**: 이메일 정규식 + 비밀번호 8자+ 검증 → `bcrypt(saltRounds=12)` → DB insert → 자동 로그인
2. **로그인**: 이메일 정규화(소문자) → `bcrypt.compare` → JWT 발급 (`HS256`, 7일) → HttpOnly·SameSite=Strict 쿠키
3. **요청 인증**: Edge Middleware가 쿠키 검증 → 통과 시 request 헤더에 `x-user-id` 주입 → 라우트 핸들러가 신뢰
4. **로그아웃**: 쿠키 만료(`maxAge=0`)

| 결정 | 근거 |
|---|---|
| Next.js App Router | 풀스택 단일 프레임워크, RSC로 초기 페이로드 절감, 미들웨어로 인증 일원화 |

| Gemini 2.5 Flash | 한국어 품질·응답 속도·무료 티어 균형. `gemini-2.0-flash`는 2026년 시점 무료 티어 차단됨 |
| Tailwind | 디자인 토큰을 클래스 형태로 유지해 디자인 시스템 일관성 확보 |
| Railway | 단일 클라우드에서 컴퓨트+영속 볼륨 동시 제공, Nixpacks 자동 빌드 |

### 6.3 의도적 비채택
| 비채택 | 이유 |
|---|---|
| Prisma/Drizzle 등 ORM | 단순 스키마 4개. ORM 오버헤드보다 prepared statement 직접 작성이 더 명확 |

## 8. 보안 및 데이터 정책

### 8.1 인증·세션
- 비밀번호 평문 저장 절대 금지. bcrypt(saltRounds=12)
- JWT 페이로드는 `userId`, `email`만 포함 (PII 최소화). 만료 7일
- 쿠키 속성: `HttpOnly`, `SameSite=Strict`, 운영 환경 `Secure`

### 8.2 입력·출력
- 모든 SQL은 **Prepared Statement** 사용 (SQL Injection 방지)
- 모든 API 엔드포인트에서 외부 입력(`req.json()`)은 타입 검증 후 사용
- 에러 응답은 사용자에게 일반 메시지만 노출, 내부 상세는 서버 로그(`console.error`)

### 8.3 인가
- 모든 대화·메시지 조회/삭제는 **userId 일치 여부**를 SQL where절에 포함
- 미들웨어 통과 후에도 라우트 핸들러가 한 번 더 소유권 검증

### 8.4 비밀 관리
- 모든 비밀(`GEMINI_API_KEY`, `JWT_SECRET`)은 환경변수
- 로컬 `.env.local` git ignore, Railway는 대시보드 Variables 사용
- 로컬 `JWT_SECRET`과 운영 `JWT_SECRET` **분리**

## 9. 성능 및 가용성

### 9.1 성능 목표
| 지표 | 목표 |
|---|---|
| 초기 페이지 로드(LCP) | ≤ 2.5초 (3G 기준) |
| 첫 토큰 수신(TTFT) | ≤ 2.0초 (Gemini 평균) |
| API 라우트 p95 | ≤ 500ms (LLM 호출 제외) |
| 빌드 결과 First Load JS | ≤ 100kB (shared chunks) |

### 9.2 가용성
- Railway 24/7 컨테이너 상시 가동
- 헬스체크: `GET /api/health` (200 OK), 실패 시 자동 재시작
- 컨테이너 재배포 시 볼륨(`/app/data`)을 통해 DB 영속 보장

### 9.3 스케일링 경로
1. **현재**: 단일 컨테이너 + SQLite + 단일 노드
2. **단기**: 컨테이너 수직 확장(CPU/메모리 상향)
3. **중기**: PostgreSQL 전환 + 컨테이너 다중화
4. **장기**: 모델 라우터 도입 (멀티 LLM), Redis 세션 캐시 검토

## 10. 운영 및 배포 전략

### 10.1 배포 파이프라인
```
GitHub push to main
   │
   ▼
Railway webhook trigger
   │
   ├─ Nixpacks detect Node 20.x (package.json engines)
   ├─ npm ci (production install)
   ├─ npm run build (next build + standalone)
   ├─ npm run postbuild (copy public + .next/static into standalone)
   ├─ Containerize & push image
   │
   ▼
Mount volume /app/data
   │
   ▼
Start: HOSTNAME=0.0.0.0 node .next/standalone/server.js
   │
   ▼
Healthcheck GET /api/health (200 OK 필요)
   │
   ▼
Promote to production traffic
```

### 10.2 환경변수 (운영)
| 변수 | 값 |
|---|---|
| `GEMINI_API_KEY` | Google AI Studio 발급 |
| `GEMINI_MODEL` | `gemini-2.5-flash` |
| `JWT_SECRET` | 32자 이상 무작위 (운영 전용) |
| `DATABASE_PATH` | `/app/data/chat.db` |
| `NEXT_PUBLIC_APP_NAME` | `AI Chat` |
| `NODE_ENV` | `production` |

### 10.3 모니터링·로깅
- 서버 로그: Railway 기본 stdout/stderr 수집
- 에러 추적: 모든 `console.error('[scope]', err)` 형태 표준화
- 향후: Sentry 또는 Logtail 연동 고려

### 10.4 백업·복구
- SQLite 파일(`/app/data/chat.db`)은 Railway 볼륨 스냅샷 정책에 위임
- 권장: 주 1회 수동 다운로드(`railway run sqlite3 ... .backup`)
